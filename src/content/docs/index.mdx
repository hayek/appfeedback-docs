## Why AppFeedback

<CardGrid>
  <Card title="One wire format" icon="list-format">
    Every platform produces the **exact same** GitHub issue body — pinned by a shared spec and a golden-fixture conformance suite that runs in every SDK's CI. Apple, Android, and Web are byte-identical.
  </Card>
  <Card title="You hold the token" icon="approve-check-circle">
    On the web, the SDK talks only to a relay **you** deploy — Cloudflare, Firebase, Appwrite, or your own. Your GitHub credential never ships to the browser.
  </Card>
  <Card title="Drop-in UI" icon="mobile-android">
    A themeable feedback sheet on Apple and a feedback widget + React component on the web — or go headless and submit from your own UI.
  </Card>
  <Card title="GitHub is the backend" icon="github">
    No new service to run. Feedback lands as a labelled GitHub issue your team already triages; the AppFeedback inbox reads it straight back.
  </Card>
</CardGrid>

## How the web relay works

<div class="afb-flow">
  <div class="node"><b>Your app</b><span>collects feedback in the widget</span></div>
  <div class="arrow">→</div>
  <div class="node"><b>Your relay</b><span>holds the token, rate-limits, verifies CAPTCHA</span></div>
  <div class="arrow">→</div>
  <div class="node"><b>GitHub issue</b><span>created in your repo, returns the number</span></div>
</div>

Native apps (Apple, Android) can talk to GitHub directly with a Keychain-held token, or use the same relay. [See the security model →](guides/security/)
